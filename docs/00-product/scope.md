# 产品范围

本文档是 Math Agent 的产品范围来源。Agent 和开发者不得从早期草稿、对话记忆或模型习惯中推断 MVP 范围；如有冲突，以本文档为准。

当前正式范围以最近产品讨论为准，而不是完全沿用早期 `product_requirments.md`。早期 PRD 仍保留为参考材料，但其中“RAG 优先、OCR/可视化后置”等旧优先级已被更新。

## 当前阶段说明

本文件仍保留 MVP 范围作为已完成和回归保护基线。MVP 主线是本地学习闭环：聊天、回答模式、OCR、可视化和本地会话。

根据 `AGENTIC_RAG_HANDOFF.md` 和 `ADR-007-agentic-rag-prototype.md`，下一阶段正式进入 **Agentic RAG Prototype / Intelligent Course Assistant Prototype**。这一阶段允许在本地优先、可验证、可回溯的前提下扩展 PDF RAG、结构化 planner、自动工具调用、引用展示、偏好和轻量记忆。

## MVP 必做

### 1. 文本聊天与数学问题输入

用户可以通过文本输入数学分析问题。输入可以包含自然语言、常见数学符号和简单公式表达，不强制要求标准 LaTeX。

MVP 重点是跑通学习对话闭环，而不是覆盖所有数学表达解析。

### 2. 回答模式控制

用户必须能够在不同回答粒度之间切换：

- 直接解答：快速给出答案和必要步骤，适合核对结果。
- 分步引导：逐步提示思路，适合学习和作业辅导。
- 仅提示：只给关键方向或下一步，适合保留独立思考空间。

默认行为不能强制所有场景都使用苏格拉底式引导。产品应允许用户快速选择“直接看答案”或“继续提示”。

### 3. 题型分类与回答策略

Agent 应识别并处理以下题型：

| 题型 | 典型问题 | 默认策略 |
| --- | --- | --- |
| 概念类 | 定义、定理、性质、区别 | 优先解释概念边界和关键反例，可使用引导式提问 |
| 计算类 | 极限、导数、积分、级数计算 | 默认分步引导，允许一键查看完整解答 |
| 证明类 | 严谨证明、命题推导、错因定位 | 默认提示证明结构和缺口，不急于铺完整证明 |
| 可视化类 | 函数图像、积分区域、空间曲面 | 触发图形生成，并配合简短解释 |

分类的目的不是追求标签本身，而是决定回答策略、是否需要可视化、是否需要引用资料。

### 4. LaTeX 输出与公式渲染

Agent 输出应支持数学公式表达，前端应能渲染常见行内公式和块级公式。

MVP 目标是让数学回答可读、可展示，不要求构建完整公式编辑器。

### 5. OCR 基础链路

OCR 是 MVP 默认输入能力之一。第一版应实现基础链路：

1. 用户上传题目或推导图片。
2. 系统识别出文本/公式内容。
3. 用户可以编辑或确认识别结果。
4. 确认后的内容进入 Agent 对话。

MVP 不要求 OCR 供应商最终锁死，也不要求识别准确率达到专业工具上限。实现上应允许后续替换 OCR provider。

### 6. 3D/函数可视化基础链路

3D 与函数可视化是 MVP 的特色亮点。第一版聚焦可实现、可演示的范围：

- 支持简单 `z = f(x, y)` 曲面。
- 支持基础 2D 函数图像或简单区域可视化。
- Agent 能判断某些题目是否适合生成可视化。
- 前端图形应支持基本交互，例如旋转、缩放或拖拽查看。

MVP 不要求覆盖复杂隐式曲面、任意积分区域自动解析或高精度数学绘图。

## MVP 可选

这些能力有价值，但不应阻塞第一版交付。

### 1. 轻量课程资料检索

可做小规模课程资料检索，用于概念解释和来源提示。第一版可以采用少量结构化条目、关键词匹配或简单检索。

不要求完整向量数据库、不要求复杂 PDF/PPT 全链路处理。

### 2. 少量知识条目

可内置少量数学分析核心概念，用于演示更稳定的概念解释和引用。

范围可以从 10-30 条开始，不必一开始整理完整 50+ 概念库。

### 3. 少量例题或往年题索引

可放入少量典型例题或往年题作为演示资源，用于“给我类似题练习”等场景。

不要求完整习题库、难度系统或个性化训练路径。

## MVP 明确不做

以下内容不进入第一版范围，除非后续明确修改本文档：

- 登录系统、用户账户和权限体系。
- 长期历史记录和跨设备同步。
- 完整个人知识库。
- 完整 RAG 平台或复杂文档管理系统。
- 大规模题库系统和个性化学习路径推荐。
- 移动端原生 App。
- 语音输入/输出。
- 社区、分享、多人协作或白板。
- 完整 CAS 级符号计算系统。
- 重型 LangGraph 或多 Agent 编排框架。
- 复杂 3D 数学建模或专业绘图软件能力。

## Agentic RAG Prototype 必做

### 1. 结构化 Agent Planner

后端必须在回答生成前产生机器可读的 planner 决策，至少包含：

- `question_type`
- `needs_retrieval`
- `needs_plot`
- `needs_clarification`
- `answer_mode`
- `memory_action`
- `reason`

Planner 输出用于驱动后续工具，而不是作为用户可见调试信息堆在 UI 上。

### 2. PDF / 课程材料 RAG

系统应支持上传课程 PDF、教材或讲义，提取文本、切分 chunk、存储文档和 chunk 元数据，并能基于用户问题检索相关片段。

第一版可以使用本地 SQLite 和简单检索或轻量 embedding provider；必须保留 provider/service 边界。

### 3. Citation-Safe Answers

当答案使用课程材料时，应展示真实来源信息，例如文件名、页码、章节或 chunk 摘要。没有检索证据时不得编造来源、页码或定理编号。

### 4. 自动工具执行

用户不应为每一轮手动选择检索、可视化、OCR、引用或记忆。Planner 应决定是否调用 retrieval、plot、clarification、profile/memory 等工具，并把结果作为回答上下文或消息附件。

### 5. 偏好和轻量记忆

系统应支持本地偏好和轻量记忆，例如默认回答模式、回答风格、自动检索/可视化开关、自定义说明、近期弱点和课程焦点。记忆注入必须有边界，不得无限增长或污染无关回答。

## Agentic RAG Prototype 不优先做

- 完整知识库管理平台。
- 多用户账户、权限、云同步。
- 专业文档标注/编辑系统。
- 大规模题库和完整个性化学习路径。
- 重型 LangGraph 或复杂多 Agent 框架，除非后续 ADR 明确修改。
- 完整 CAS 或专业 3D 建模。

## 回答模式矩阵

回答行为由“题型”和“回答模式”共同决定。

| 回答模式 | 适合场景 | 输出要求 |
| --- | --- | --- |
| 直接解答 | 用户明确想快速得到答案或核对结果 | 给出结论、关键步骤和必要说明 |
| 分步引导 | 用户正在学习、作业卡住或未明确要求答案 | 给出理解确认、关键提示和下一步行动 |
| 仅提示 | 用户希望保留思考空间 | 给最小必要提示，不展开完整过程 |

默认建议：

- 概念类：默认分步讲解，可穿插反例。
- 计算类：默认分步引导，提供直接答案入口。
- 证明类：默认仅提示或分步引导，按用户要求再展开完整证明。
- 可视化类：优先生成图形，并用短说明解释图形含义。

## 成功标准

第一版产品成功不以功能数量衡量，而以是否形成可用学习闭环衡量：

- 用户能输入文本或图片题目。
- 用户能选择或切换回答模式。
- Agent 能根据题型给出合适粒度的回答。
- 常见公式能正常展示。
- 至少一类 3D/函数图形能被生成并用于解释。
- RAG、题库和个人知识库即使暂未完成，也不会影响核心闭环。
