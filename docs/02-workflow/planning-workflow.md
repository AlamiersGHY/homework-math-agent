# 需求方案制定流程

本文档定义当用户提出模糊功能想法、UI 想法、产品改进或方案制定请求时，Agent 如何把想法快速收敛为可执行方案。

本流程的目标不是生成冗长 PRD，而是让 Agent 用较低用户介入完成：理解意图、检查上下文、明确边界、给出推荐方案、形成执行计划，并判断是否需要更新 SDD 状态或 ADR。

## 触发来源

本文件本身不会自动触发。真正的触发入口是 `AGENTS.md` 中的 Mandatory Context Routing。

当 `AGENTS.md` 识别到任务属于模糊需求、UI 想法、产品改进、功能规划或方案制定时，Agent 必须读取并执行本文件。

## 何时进入本流程

进入本流程的典型情况：

- 用户说“我想做一个功能但还没想清楚”。
- 用户说“帮我设计一个 UI”。
- 用户说“你帮我规划一下”。
- 用户询问“这个需求应该怎么做”。
- 需求可能影响产品行为、Agent 行为、API、架构、依赖、状态管理或长期方向。
- 用户给出方向，但希望 Agent 主动补齐行业做法、实现路径和边界。

不需要进入完整流程的情况：

- 明确的小 bug 修复。
- 文案微调。
- 已有 plan 的直接实现。
- 不影响长期方向的小范围代码整理。
- 用户明确要求只回答一个局部问题。

## 默认工作原则

- 轻量自动触发：非平凡且不清楚的需求先规划，明确小任务可以直接执行。
- 本地优先：先读 SDD、代码、已有决策和当前状态，再考虑外部参考。
- 低介入：最多问 1-3 个会改变方案的关键问题。
- 推荐优先：Agent 应给出明确推荐，而不是只罗列选项。
- 可交付优先：方案必须能在当前阶段形成可验证的交付。
- 少增系统：不新增 feature brief 目录；方案状态优先进入 `active.md`，完成后简要归档到 `completed.md`。
- 默认不用子 Agent：只有复杂调研、多路线并行分析或用户明确要求时才建议使用。

## 流程步骤

### 1. Context Check

Agent 先完成可发现信息收集，不把可通过仓库确认的问题丢给用户。

至少检查：

- `AGENTS.md`
- `docs/INDEX.md`
- `docs/04-logs/active.md`
- 与任务相关的 product、architecture、workflow、ADR、eval 或代码文件

如涉及 UI，应检查前端结构、技术栈和已有组件边界。如涉及后端或 Agent 行为，应检查 API 契约、service/provider 边界和 evals。

### 2. Intent Snapshot

Agent 将用户想法压缩成一句清楚的目标：

- 用户想要什么。
- 解决哪个痛点。
- 面向哪个使用场景。
- 本轮最低成功标准是什么。

### 3. Scope Boundary

Agent 明确：

- 本轮做什么。
- 本轮不做什么。
- 是否属于当前 MVP 范围。
- 是否触碰产品范围、架构、API、依赖或长期方向。

如果需要扩大范围，进入 Decision Gate，不直接实现。

### 4. Design Direction

Agent 给出推荐设计方向。

UI 需求应说明：

- 信息架构。
- 核心交互。
- loading、error、empty 等必要状态。
- 视觉方向。
- 明确不做的装饰或复杂效果。

后端、API 或 Agent 行为需求应说明：

- 主要数据流。
- API 或 schema 是否变化。
- service/provider 边界。
- 失败状态和验证方式。

如果存在重要取舍，给出 2-3 个候选方案并标注推荐项。

### 5. Ask Only Critical Questions

只有当答案会改变方案时才询问用户。

问题应满足：

- 最多 1-3 个。
- 尽量给出推荐选项。
- 不询问可通过仓库或文档发现的问题。
- 如果没有关键问题，直接列出默认假设。

### 6. Output Plan

规划输出默认分两层：先给短卡片，再给执行计划。

第一层是方案卡片，让用户快速看懂“到底要做什么”：

```md
## 方案卡片

- 要做什么：一句话说明最终形态。
- 为什么做：对应的用户痛点或项目目标。
- 做到什么程度：本轮 MVP 边界。
- 不做什么：避免范围膨胀。
- 推荐方案：Agent 的明确建议。
- 需要你确认：最多 1-3 个关键问题；如果没有关键问题，直接给默认假设。
```

第二层是执行计划：

```md
## 执行计划

- Key Changes：要改哪些模块或能力。
- Data/API/UI Impact：是否影响接口、状态、数据流或 UI 结构。
- Test Plan：如何验证。
- SDD Sync：active/completed/ADR/evals 是否需要更新。
- Assumptions：默认采用的假设。
```

对于很小的规划任务，可以只给方案卡片和简短执行计划。对于复杂任务，执行计划可以展开，但仍应避免写成长 PRD。

## Decision Gate

规划结束前，Agent 必须判断规划结果应该进入哪里：

| 情况 | 记录位置 |
| --- | --- |
| 只是当前任务方案 | `docs/04-logs/active.md` |
| 完成一个可交付单元 | `docs/04-logs/completed.md` |
| 产生未来任务必须遵守的长期约束 | `docs/03-decisions/` ADR |
| 影响产品范围、优先级或非目标 | `docs/00-product/scope.md`，必要时 ADR |
| 影响 Agent 行为、分类、回答策略或可视化触发 | `evals/` |
| 只是临时实现细节 | 不进入 ADR |

`active.md` 是当前工作台，`completed.md` 是交付日志，`decisions/` 是长期约束。不要让三者互相替代。

## 外部参考规则

外部调研不是默认第一步。

只有在以下情况才建议外部查找资料或参考行业做法：

- UI 类型没有现有项目模式可参考。
- 用户明确要求参考优秀实践。
- 技术方案涉及当前 docs 未覆盖的 provider、库或平台。
- 本地信息不足以做出可靠推荐。

外部参考必须服务于当前方案，不应变成泛泛资料堆叠。

## 完成标准

一次规划完成时，应该能回答：

- 本轮要做的东西大概长什么样。
- 为什么值得做。
- 做到什么程度就算完成。
- 哪些明确不做。
- 需要改哪些模块或文档。
- 如何验证。
- 是否需要 ADR、eval 或 scope 更新。
- 用户还需要确认什么；如果不确认，默认假设是什么。
