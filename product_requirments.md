# 数学分析启发式学习助手 - 产品需求文档

## 1. 产品定位

一款面向工科本科生的数学分析课程智能学习助手，通过"引导式提问"和"启发式思考"机制，帮助学生深入理解概念、掌握解题方法，而非直接提供答案。

**核心价值主张**：

- **精准引导**：解决传统LLM回答"过长过散"、有效信息密度低的痛点
- **启发式教学**：采用苏格拉底式提问法，培养学生独立思考能力
- **学科专注**：深耕数学分析单一领域，提供专业化、结构化的知识支持

---

## 2. 目标用户

**主要用户群体**：正在学习数学分析课程的工科本科生

**典型用户画像**：

- 大一/大二学生，刚接触严格的数学证明和ε-δ语言
- 课后自学时需要概念澄清和解题指导
- 希望理解"为什么这样做"而不仅是"怎么做"
- 对长篇大论的理论解释耐心有限，偏好结构化、可操作的提示

**使用场景**：

1. **概念澄清**：课堂上没听懂的定理，课后询问Agent帮助理解
2. **作业辅导**：遇到不会做的习题，寻求思路引导而非直接答案
3. **考前复习**：针对薄弱知识点进行针对性练习和诊断
4. **错题分析**：上传自己的错误推导，让Agent指出逻辑漏洞

## 3. 功能需求清单

### M1: 核心对话Agent（P0 - MVP必须实现）

#### 1.1 自然语言提问接收

- **功能描述**：用户通过文本输入数学问题，支持日常语言表达
- **输入格式**：纯文本，可选包含简单数学符号（如 x^2, sum, integral等）
- **约束条件**：
  - 不支持图片/手写公式识别（M5模块，二期规划）
  - 不强制要求LaTeX输入语法（降低用户使用门槛）
- **验收标准**：能正确解析90%以上的常见数学问题表述

#### 1.2 问题分类与路由

- **功能描述**：自动识别问题类型，执行差异化引导策略
- **分类体系**：
  - **概念类**（Conceptual）：定义、定理、性质相关的理解性问题
    - 示例："一致连续和普通连续有什么区别？"
    - 策略：苏格拉底式提问，引导学生自己发现差异
  - **计算类**（Computational）：需要具体数值或表达式求解的问题
    - 示例："求lim(x→0) sin(x)/x的值"
    - 策略：分步验证，让学生自己完成每一步计算
  - **证明类**（Proof）：需要逻辑推理和严谨论证的问题
    - 示例："证明闭区间上连续函数必有最大值"
    - 策略：漏洞定位，指出推理链条中的断层
- **技术实现**：LLM基于Prompt进行分类判断，输出分类标签
- **验收标准**：分类准确率 > 85%（人工抽样评估）

#### 1.3 引导式回复生成

- **功能描述**：根据问题类型生成结构化、精简的引导性回复
- **输出约束**：
  - **长度限制**：单次回复 ≤ 300字（概念解释）或 ≤ 150字（单步提示）
  - **结构要求**：
    ```
    [理解确认] 回应用户当前的理解状态
    [引导提问] 提出一个启发性问题或关键提示
    [行动建议] 给出明确的下一步操作指引
    ```
  - **禁止项**：
    - 不使用"首先...其次...最后..."的长篇结构
    - 不直接给出完整解答（除非触发降级机制）
    - 不包含冗余的客套话和重复解释
- **渐进式提示机制**：
  - 同一问题最多提供3轮渐进式提示
  - 第1轮：思路方向提示（最抽象）
  - 第2轮：关键步骤提示（中等粒度）
  - 第3轮：具体方法提示（较详细）
  - 第4轮及以上：提供完整解答，但标注"参考答案"
- **会话历史管理**：
  - 记忆最近5轮对话上下文
  - 避免重复提问或矛盾的建议
  - 超出窗口的历史进行摘要压缩
- **验收标准**：
  - 对10个测试问题的回复符合"引导式"原则
  - 平均响应时间 < 3秒（不含LLM调用延迟）
  - 用户满意度评分 > 4/5（主观评估）

#### 1.4 LaTeX公式渲染支持

- **功能描述**：Agent输出的数学公式在前端自动渲染为美观格式
- **技术要求**：
  - 支持行内公式 `$...$` 和块级公式 `$$...$$`
  - 覆盖数学分析常用符号：积分∫、求和∑、极限lim、偏导∂等
  - 移动端和桌面端显示正常，无布局溢出
- **实现方案**：前端集成KaTeX库（轻量、快速、兼容性好）
- **验收标准**：所有测试公式正确渲染，无错位或乱码

### M2: 专业知识库检索增强（P1 - 应该有，MVP可实现简化版）

#### 2.1 核心概念索引

- **功能描述**：构建数学分析核心知识的结构化索引
- **内容范围**：
  - 50个核心概念（极限、连续、微分、积分、级数等章节的关键定义和定理）
  - 每个概念包含：
    - 标准定义（精确表述）
    - 关键定理（相关的重要结论）
    - 经典反例（帮助理解边界条件）
    - 教材出处（章节/页码，便于学生查阅纸质书）
- **数据来源**：
  - 主流《数学分析》教材（如华东师大版、复旦版）
  - MIT OpenCourseWare开放资源
  - 教师整理的常见误区总结
- **存储形式**：JSON结构化数据 + 向量嵌入（用于语义检索）

#### 2.2 RAG检索链路

- **功能描述**：根据用户问题召回相关知识片段，增强Agent回答的准确性
- **工作流程**：
  1. 用户提问 → 生成查询向量
  2. 向量数据库检索 → Top-3相关知识片段
  3. 将知识片段注入Prompt上下文
  4. LLM基于增强上下文生成回答
- **技术难点预判**：
  - 向量相似度匹配的准确性（可能需要调整embedding模型）
  - 检索结果的相关性过滤（避免无关信息干扰）
- **MVP简化方案**：
  - 初期可使用关键词匹配代替向量检索（降低复杂度）
  - 仅对"概念类"问题启用RAG，计算/证明类问题依赖LLM自身知识
- **验收标准**：
  - 概念类问题的回答中，正确引用来源的比例 > 80%
  - 检索响应时间 < 500ms

#### 2.3 引用标注

- **功能描述**：在Agent回答中标注信息来源，增强可信度
- **标注格式**：

  ```
  根据《数学分析》（华东师大版）第3章第2节，一致连续的定义是...

  [来源：教材 P78, 定理3.2.1]
  ```

- **约束条件**：
  - 如果知识库中无相关信息，明确告知用户"该知识点暂未收录"
  - 不编造虚假的引用来源
- **验收标准**：所有引用均可追溯到知识库中的具体条目

### M3: 习题库系统（P2 - 长期规划，MVP暂不实现）

#### 3.1 习题提供与抽取

- **功能描述**：根据用户需求提供针对性练习题
- **抽题策略**：
  - 按知识点抽取（用户指定"我想练ε-δ语言的题目"）
  - 按难度递进（从易到难排列）
  - 随机抽取（综合复习模式）
- **MVP替代方案**：可在Prompt中硬编码10-20道经典例题，作为临时方案

#### 3.2 题目标注与格式化

- **元数据结构**：
  ```json
  {
    "id": "limit_001",
    "question": "求lim(x→0) (e^x - 1)/x",
    "difficulty": 2,
    "tags": ["极限", "洛必达法则"],
    "solution": "...",
    "hints": ["提示1", "提示2", "提示3"],
    "common_mistakes": ["错误1", "错误2"]
  }
  ```

#### 3.3 针对性训练引导

- **功能描述**：针对考试题型的专项训练模式
- **实现难度**：高（需要大量题目数据和精细的引导逻辑）
- **决策**：放入二期规划

### M4: 风格切换（P1 - 应该有，技术难度低）

#### 4.1 三种回复风格

- **严格模式**（Strict）：
  - 语气严肃，强调严谨性
  - 对用户的不严谨表述立即纠正
  - 适合备考冲刺阶段
- **鼓励模式**（Encouraging，默认）：
  - 语气温和，多用正向反馈
  - 即使学生犯错也先肯定再指正
  - 适合初学者建立信心
- **简洁模式**（Concise）：
  - 极简表达，只给最关键的信息
  - 省略所有客套话和解释性文字
  - 适合复习阶段快速回顾

#### 4.2 切换机制

- **命令方式**：用户输入 `/style strict` 等命令切换
- **持久化**：风格选择保存到用户会话状态
- **默认值**：首次使用默认为"鼓励模式"

#### 4.3 技术实现

- 在System Prompt中添加风格指令模板
- 根据用户选择动态替换Prompt中的语气描述部分
- **验收标准**：风格切换即时生效，三次风格的回复有明显语气差异

---

### M5: OCR图片/手写公式识别（P1 - 应该有，但技术难度中高）

#### 5.1 功能描述

- 用户上传图片（拍照的题目或手写推导），Agent识别其中的数学内容并作答
- **价值**：大幅降低输入门槛，提升用户体验

#### 5.2 技术方案预判

- **方案A**：调用商用OCR API（如Mathpix、百度OCR）
  - 优点：准确率高，开发成本低
  - 缺点：需要付费，依赖外部服务
- **方案B**：开源模型自建（如Pix2Text、Nougat）
  - 优点：可控性强，无持续费用
  - 缺点：部署复杂，需要GPU资源，准确率需调优

#### 5.3 MVP决策

- **建议**：MVP阶段先实现文本输入，OCR作为"快速跟进"功能（MVP完成后1-2周内补充）
- **原因**：
  - OCR不是核心功能的必要条件
  - 技术风险较高，可能拖延整体进度
  - 可以先收集用户反馈，验证产品价值后再投入

#### 5.4 验收标准（若实现）

- 印刷体公式识别准确率 > 90%
- 手写公式识别准确率 > 70%（取决于书写工整度）
- 端到端处理时间 < 5秒

---

### M6: 图形可视化（P2 - 长期规划）

#### 6.1 功能描述

- 绘制函数图像、几何示意图，辅助直观理解
- 示例：绘制f(x)=sin(x)/x的图像，观察x→0时的趋势

#### 6.2 技术方案

- 前端：D3.js或Chart.js（交互式图表）
- 后端：Matplotlib生成静态图，或SymPy生成交互式Plot

#### 6.3 MVP决策

- **放入二期规划**：非核心功能，开发成本高，收益相对有限

## 4. 非功能性需求

### 4.1 性能要求

| 指标           | 目标值             |
| -------------- | ------------------ |
| 端到端响应时间 | < 5秒（含LLM调用） |
| 知识库检索时间 | < 500ms            |
| 前端首屏加载   | < 2秒              |
| 并发用户支持   | ≥ 50人同时在线     |

### 4.2 可用性要求

- 无需注册即可试用基础功能
- 高级功能（如历史记录保存、风格定制）需登录
- 支持主流浏览器（Chrome、Firefox、Safari、Edge）

### 4.3 可维护性要求

- 所有Prompt模板外部化管理（JSON/YAML文件），支持热更新
- 代码注释覆盖率 > 60%
- 核心模块单元测试覆盖率 > 80%

### 4.4 可扩展性要求

- 新增学科只需添加知识库数据，无需修改核心Agent逻辑
- 预留`subject`字段作为元数据，支持未来扩展到其他数学课程

### 4.5 成本控制

- 单次对话LLM成本 < $0.01（使用GPT-5.4-mini或等价模型）
- 月度运营成本（含服务器、API调用）< $50

## 7. 成功标准

### 7.1 产品层面

- ✅ 完成M1全部功能和M2简化版

- ✅ 能够作为完整作品提交（有演示、有文档、有线上版本）

### 7.2 工程层面

- ✅ 验证Harness Engineering方法论的有效性

  - 形成的Harness文档是否可复用到下一个项目？
- ✅ 代码质量达标
  - 无严重bug
  - 测试覆盖率 > 60%
  - 代码结构清晰，易于维护

### 7.3 个人成长层面

- ✅ 掌握LangGraph Agent开发流程
- ✅ 理解RAG系统的搭建和优化
- ✅ 积累全栈开发经验（前后端联调、部署运维）
