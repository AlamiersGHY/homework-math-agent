# 反馈处理流程

本文档定义当用户、开发者或 Agent 发现问题时，如何分类、决策、执行和记录。目标是让反馈进入 SDD 系统，而不是散落在聊天记录里。

## 反馈来源

反馈可以来自：

- 用户试用。
- 开发者手测。
- 自动化测试失败。
- eval 失败。
- 部署或运行日志。
- Agent 在实现中发现的风险。

## 反馈分类

收到反馈后，先归类：

| 类型 | 说明 | 默认处理 |
| --- | --- | --- |
| Bug | 已承诺功能不工作 | 优先修复 |
| UX 问题 | 功能可用但不顺手 | 记录并评估优先级 |
| Agent 行为问题 | 回答模式、分类、提示策略不符合预期 | 更新 eval，再修行为 |
| Product Change | 反馈要求改变范围或优先级 | 需要用户决策，可能更新 scope |
| Architecture Change | 需要改变技术路线、依赖或 API | 需要 ADR |
| Tech Debt | 当前可接受但未来需要处理 | 记录到 tech debt |
| Question | 反馈本身不清楚 | 询问澄清 |

## 决策流程

Agent 应按以下顺序处理反馈：

1. 判断反馈是否可复现或可描述。
2. 判断是否属于当前 MVP 范围。
3. 判断是否需要改变产品、架构、API 或依赖。
4. 如果不需要决策，制定最小修复任务。
5. 如果需要决策，标记 `Needs Decision`，给出候选方案和推荐方案。
6. 如果属于未来工作，记录到 `tech-debt-tracker.md`。

## 执行规则

### Bug

处理方式：

- 先写或更新能暴露问题的测试/eval。
- 再修复实现。
- 最后运行相关验证。

如果无法建立自动验证，必须说明手动复现与验证步骤。

### UX 问题

处理方式：

- 判断是否影响 MVP 主链路。
- 若影响输入、回答模式、OCR、可视化或部署演示，优先处理。
- 若只是体验增强，记录到 tech debt 或下一阶段。

### Agent 行为问题

处理方式：

- 先把反馈转成 eval case。
- 明确期望行为和禁止行为。
- 再修改 prompt、policy 或 service。
- 运行 eval 或记录无法运行的原因。

### Product Change

处理方式：

- 不直接实现。
- 检查 `scope.md`。
- 如果确实改变范围，先更新 `scope.md` 或让用户确认。

### Architecture Change

处理方式：

- 不直接实现。
- 检查已有 ADR。
- 新增或更新 ADR 后再动代码。

## 记录位置

- 当前正在处理：`docs/04-logs/active.md`
- 已完成反馈：`docs/04-logs/completed.md`
- 延后问题：`docs/04-logs/tech-debt-tracker.md`
- 产品范围变化：`docs/00-product/scope.md`
- 架构决策变化：`docs/03-decisions/`
- Agent 行为变化：`evals/`

## 完成标准

反馈处理完成时，必须能回答：

- 反馈属于什么类型。
- 是否在当前 MVP 范围内。
- 做了什么处理。
- 如何验证。
- 是否还有残留风险。
- SDD 状态是否已经同步。
