# 产品路线图

本文档定义 Math Agent 的阶段性交付路线。路线图服务于短周期 AI Coding：每个阶段都应形成可验证的产物，而不是只完成抽象设计。

## Phase 0：SDD 基座与 Eval 基座

目标：让后续 AI Coding 有稳定上下文、文档入口和行为验收锚点。

交付物：

- `AGENTS.md` 全局工作协议。
- `docs/` SDD 文档骨架。
- `evals/` 行为评估入口。
- 产品范围、架构方向、测试策略的最小可执行文档。

完成标准：

- 新 Agent 能通过 `AGENTS.md`、`docs/INDEX.md`、`docs/04-logs/active.md` 找到当前任务和产品范围。
- MVP 范围不再只存在于对话记忆中。

## Phase 1：最小对话 Agent 与回答模式

目标：跑通文本输入到数学回答的最小闭环。

核心能力：

- 文本问题输入。
- 题型分类：概念类、计算类、证明类、可视化类。
- 回答模式：直接解答、分步引导、仅提示。
- LaTeX 风格数学输出。
- 基础多轮上下文或提示轮次状态。

完成标准：

- 用户能完成一次可用数学问答。
- 用户能切换“直接答案”和“引导提示”。
- Agent 行为有初始 eval cases 覆盖。

## Phase 2：OCR 输入链路

目标：降低数学题输入门槛，让图片题目能进入对话流程。

核心能力：

- 图片上传入口。
- OCR provider 抽象。
- OCR 识别结果展示。
- 用户可编辑确认识别结果。
- 确认后的文本进入 Phase 1 对话链路。

完成标准：

- 上传图片后能得到可编辑文本。
- OCR 失败时有明确降级提示。
- OCR provider 可以后续替换，不写死在业务流程中。

## Phase 3：3D/函数可视化链路

目标：形成项目特色亮点，用图形增强函数、积分和空间曲面的理解。

核心能力：

- Agent 判断题目是否适合可视化。
- 生成基础 plot spec。
- 前端渲染 2D 函数或 `z = f(x, y)` 曲面。
- 图形支持基础交互查看。
- 回答中配合解释图形代表的数学意义。

完成标准：

- 至少能稳定演示一类 3D 曲面图。
- 至少能稳定演示一类 2D 函数或区域图。
- 可视化触发行为有 eval cases 覆盖。

## Phase 4：轻量 RAG / 题库增强

目标：在不拖慢核心闭环的前提下，提高概念解释和练习推荐的专业性。

核心能力：

- 少量课程知识条目。
- 简单课程资料检索或关键词匹配。
- 概念解释中的来源提示。
- 少量例题或往年题索引。

完成标准：

- 概念类问题能引用已收录资料。
- 未收录资料时不编造来源。
- 检索增强失败时不影响基础回答。

## Phase 5：联调、部署、试用反馈与项目总结

目标：把项目变成可访问、可演示、可复盘的交付物。

核心能力：

- 前后端联调。
- 基础错误处理和加载状态。
- 云平台临时部署。
- 小规模真实用户试用。
- 记录产品反馈和 AI Coding / SDD 实践总结。

完成标准：

- 有线上可访问版本。
- 有演示路径和基础使用说明。
- 记录关键问题、技术债和后续改进方向。

## 里程碑取舍原则

- 每个阶段都应保持可运行或可验证。
- OCR 和可视化是核心体验，不应无限后置。
- RAG、题库和个人知识库是增强能力，不应阻塞核心闭环。
- 如果开发周期紧张，优先保留“输入 -> 回答模式 -> 可视化/OCR 亮点 -> 可部署演示”的主线。
