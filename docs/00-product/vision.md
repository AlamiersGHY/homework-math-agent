# 产品愿景

本文档是 Math Agent 的产品方向来源，用来说明项目为什么存在、服务谁、核心价值是什么。具体 MVP 边界以 `scope.md` 为准，阶段计划以 `roadmap.md` 为准。

## 产品定位

Math Agent 是一个面向数学分析学习场景的 AI 学习助手，不是通用数学聊天机器人。

它的目标不是覆盖所有数学问题，也不是单纯替代搜索或答案生成工具，而是在数学分析学习中提供更合适的“教学节奏”：用户需要答案时可以快速获得完整解答，用户需要学习时可以获得逐步提示、概念澄清和空间直观辅助。

在完成本地 MVP 闭环后，下一阶段产品方向是 **Agentic RAG Prototype / Intelligent Course Assistant Prototype**。产品应逐步从“能回答数学问题的学习工作台”升级为“围绕课程资料、课堂讲义和用户学习状态工作的智能课程助教”：用户自然提问，系统自动判断是否需要检索课程材料、引用来源、生成图形、澄清问题或调整讲解风格。

## 目标用户

主要用户是正在学习数学分析课程的工科本科生，尤其是：

- 大一/大二刚接触严格证明、极限语言和积分思想的学生。
- 课后自学时需要概念澄清、解题思路和错因定位的学生。
- 面对长篇解释容易失去耐心，更偏好短、结构化、可执行提示的学生。
- 在多元函数、积分区域、空间曲面等内容中缺少空间想象力的学生。

## 核心价值

### 1. 控制讲解粒度

传统 LLM 经常一次性给出过长、过散的回答。Math Agent 的核心价值之一是根据用户意图控制帮助粒度：

- 想快速核对结果时，允许直接解答。
- 想练习思路时，给出分步引导。
- 想保留独立思考空间时，只给关键提示。

产品不强制所有问题都使用苏格拉底式回答，而是让回答模式服务于当前学习场景。

### 2. 支持启发式学习

对于概念理解、证明构造和计算推导，Agent 应优先帮助用户发现关键条件、常见误区和下一步行动，而不是机械铺满完整过程。

尤其在证明题中，产品应强调：

- 定位推理链条的缺口。
- 提醒需要使用的定义、定理或反例。
- 帮助用户自己补上关键步骤。

### 3. 降低题目输入门槛

数学分析问题经常来自纸质教材、PPT、手写推导或截图。OCR 是产品默认输入能力的一部分。

MVP 不追求完美识别，但应提供“图片输入 -> OCR 识别 -> 用户可编辑确认 -> 进入 Agent”的基础链路，避免 OCR 错误直接污染后续解题。

### 4. 用可视化增强空间直觉

函数图像、积分区域和三维空间曲面是数学分析学习中的高价值辅助对象。产品应把 2D/3D 可视化作为特色亮点，而不是远期装饰功能。

第一阶段可视化重点不是覆盖所有数学图形，而是优先服务：

- `z = f(x, y)` 类型空间曲面。
- 常见二元函数的直观形状。
- 简单积分区域或边界的视觉理解。

### 5. 用课程资料提高可信度

进入 Agentic RAG Prototype 后，Math Agent 的回答不应只依赖通用模型记忆。对于定义、定理、课堂讲义和教材相关问题，系统应优先尝试从用户上传或本地收录的课程资料中寻找依据，并在有证据时展示真实来源。

这一价值的重点是可靠性，而不是堆叠更多知识。没有可用资料时，Agent 应明确说明缺少来源支持，不能编造文件名、页码、章节或 citation。

## 产品原则

- 可交付优先：先完成能真实使用的闭环，再扩展功能广度。
- 少功能但有亮点：宁可少做泛化功能，也要把回答模式、OCR、可视化做成能展示的核心体验。
- 教学节奏控制优先于知识堆叠：产品差异点是“怎么教”，不是只看“知道多少”。
- 用户可控优先：用户应能快速切换直接答案与引导提示，不被默认教学风格绑死。
- Chat-first 优先：自动检索、绘图、引用和记忆应融入对话流程，不把界面变成需要用户手动选择每个工具的工作台。
- 来源可靠优先：课程材料引用必须来自真实检索结果，缺少证据时不伪造来源。
- 轻量工程优先：功能设计必须服务短周期 AI Coding，不以完整平台化为第一目标。

## 非目标提醒

Math Agent 当前不是：

- 完整在线教育平台。
- 个人知识库系统。
- 大规模题库训练系统。
- 通用数学 CAS 工具。
- 专业绘图软件。
- 完整个人知识库平台。
- 长期学习路径推荐系统。
- 重型多 Agent 编排平台。

这些方向可以作为后续扩展，但不应干扰当前交付目标。
